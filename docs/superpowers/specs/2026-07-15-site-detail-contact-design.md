# 官网详情页与项目联系体验设计

## 背景

现有官网的大框架、首页和导航已建立清晰的品牌叙事，但多个详情页面继续使用同一组大体量、静态的介绍区块。编号列表只突出第一项，右侧文案没有随选择变化，读者难以理解二至四项与内容的对应关系。页面底部的圆形“项目咨询”按钮与顶部入口、联系页含义重复，且联系页没有可提交的项目需求表单。

用户要求保留官网的总体框架，在新的隔离 worktree 中改善详情内容的阅读连续性，并在下次验收时提供结果。

## 目标

- 让详情页重复的“能力、场景、交付”信息以连续、可切换的方式呈现，减少逐段滚动造成的信息割裂。
- 在桌面端提供编号 01-04 的悬停、焦点和点击切换；在触屏端提供等价的点击展开体验。
- 收敛“关于光泰”页面的后续大区块，使核心能力与交付逻辑可以在一个连续章节中理解。
- 将“光泰 AI 应用工厂”明确为平台介绍，缩短页面并预留“前去体验”按钮，不配置外部跳转。
- 将解决方案菜单第一组升级为“AI 与具身智能”，体现 AI 能力、智能体、AI 工作流、知识库/自动化及具身智能，同时与 AI 应用工厂明确分工。
- 将顶部和页底联系入口统一到项目联系表单，保留邮箱和 Webhook 的空配置位，不宣称没有渠道时已经提交成功。

## 非目标

- 不改变首页 Hero、Spline 场景、主视觉资产或首页业务区块顺序。
- 不创建新的虚构解决方案路由，不调整既有 SEO URL。
- 不接入未提供的收件邮箱、企业微信或飞书 Webhook，不把项目需求发送到第三方服务。
- 不改动生产部署、Docker 或环境密钥。

## 信息架构

### 详情内容交互

新增可复用的“编号内容切换器”组件，接收四项（或更少）结构化内容。

1. 桌面端左侧显示稳定的 01-04 垂直索引。当前项数字、标题与细线放大或高亮；右侧仅展示当前项的标题、说明和关联视觉。
2. 鼠标悬停、键盘 Tab 聚焦、Enter/Space 点击任意索引，都会更新当前项。点击后状态保持，避免鼠标离开时跳回第一项。
3. 使用 `button` 与 `aria-pressed` 或关联的 `tab`/`tabpanel` 语义，屏幕阅读器能够获取当前项；不以仅 hover 的 CSS 实现。
4. 移动端采用纵向按钮列表和内容面板，触屏点击可切换，内容不依赖悬停，且不出现横向溢出。

此组件替代目前详情页中“左侧编号 + 右侧静态列表”的阅读断点。交付路径仍保留为四步，但缩短单项高度、补足顺序关系；场景卡片按页面需要保留，避免把所有内容硬塞进一屏。

### 关于光泰

首个“定位与价值”章节保留。后续页面不再让多个大型独立区块平铺，而将能力、场景和交付串为“光泰如何把平台、软件、空间与设备组织为项目”的连续叙事：

- 能力切换器说明平台、软件集成、空间智能和具身设备四个层次；
- 相邻的交付步骤以紧凑顺序承接，不重复展开同一套能力；
- 保留现有真实素材与页面锚点，减少视觉和信息的断层。

### AI 应用工厂与解决方案

AI 应用工厂页面只回答“平台是什么、如何把模型和工具组织为可交付底座”。内容收敛为平台能力、典型交付方式和一个未配置目标的“前去体验”按钮。

解决方案菜单第一组改为“AI 与具身智能”。该组保留既有 AI 能力与具身智能页面链接，并在组说明和预览中覆盖：智能体、AI 工作流、知识库与 RAG、工具调用/自动化、AI 编程和实体设备协同。页面职责如下：

| 页面             | 主要回答的问题                                         |
| ---------------- | ------------------------------------------------------ |
| AI 应用工厂      | 光泰以什么平台底座接入模型、知识、工具和数据。         |
| AI 能力解决方案  | 如何把智能体、工作流、知识库和自动化嵌入具体客户流程。 |
| 具身智能解决方案 | 如何让 AI 调度和协同实体设备进入物理空间。             |

这样不需要新增空路由，也能避免“AI 应用工厂”和“AI 能力”把同一段平台说明重复两次。

### 项目联系

- 顶部 CTA 文案改为“项目联系”，导航中的“联系我们”仍作为普通信息入口；两者都落到 `/contact`。
- 所有非联系页的底部联系区改为轻量的“提交项目需求”入口，移除突兀的圆形按钮。
- `/contact` 以真实项目表单作为主内容，字段为姓名、单位、联系方式、项目类型、计划时间和需求描述；必填项在客户端校验。
- 新增中央配置模块读取公开的联系渠道配置。当前邮箱和 Webhook 均为空：表单通过校验后显示“联系渠道尚未配置，请稍后联系”的明确状态，不显示虚假的“提交成功”。以后配置邮箱或 Webhook 时，只修改配置和服务端/边缘端投递实现，不改表单 UI。

## 数据流与失败处理

页面内容继续由 Astro Content Collections 的 JSON 进入 `SitePage`，编号切换器从页面能力或场景数据生成。交互仅在浏览器内更新已存在的内容，不发起网络请求。

表单提交先执行浏览器原生及自定义校验。渠道配置为空时，阻止请求并显示配置状态；配置了受支持渠道后才调用投递适配器。网络或投递失败必须显示失败状态，保留已填写的内容供用户修改，绝不显示成功提示。

## 验收标准

- 桌面端 01-04 可通过悬停、Tab 聚焦和点击切换，当前编号及右侧内容同步更新。
- 移动端仅以点击切换，不需要 hover，且 375px 宽度没有横向滚动。
- 关于光泰的后续内容读起来连续，核心说明不被分散到多个大区块中。
- AI 工厂页面明显短于现状，且有不跳转的“前去体验”预留按钮。
- 解决方案第一组显示“AI 与具身智能”，AI 内容不与工厂平台介绍重复。
- 顶部、页底和联系页的项目联系入口均指向同一表单；空渠道配置不会出现伪成功。
- 格式、类型、内容、构建、渲染和浏览器测试通过；需单独记录任何既有的不稳定测试。

## 风险

- 静态站点在没有邮件服务、Webhook 或后端的情况下不能真实投递表单。当前以清晰的未配置状态处理，待用户提供渠道后再接入投递。
- 统一详情组件不能强行替代所有页面的业务叙事。若某个页面的内容无法合理映射到四项编号，应保留专属章节而非填充泛化文案。
- Astro ClientRouter 持久化头部下，表单和编号切换器需要在路由切换后重新初始化，测试应覆盖首访与无刷新切页两种情况。
